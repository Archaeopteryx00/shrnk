# UI_STYLE.md

## Purpose

This document defines the visual identity, user experience, and interface guidelines for Shrnk.

Its purpose is to ensure every page, component, and interaction follows one consistent design language.

Shrnk should feel like a real SaaS product rather than a university project or an AI-generated template.

---

# Design Philosophy

Shrnk is a developer-focused productivity tool.

The interface should communicate:

- Fast
- Lightweight
- Professional
- Minimal
- Technical
- Premium

The UI should disappear into the background and allow users to focus on managing links.

Every design decision should prioritize clarity over decoration.

---

# Design Inspirations

Primary references:

- Dub.co
- Vercel Dashboard
- Linear
- Clerk
- Better Auth
- GitHub
- Raycast

The application should resemble modern developer tools rather than consumer applications.

---

# Avoid

Never generate interfaces that resemble:

- AI SaaS templates
- ChatGPT clones
- Crypto dashboards
- NFT websites
- Neon cyberpunk themes
- Heavy glassmorphism
- Colorful gradients
- Excessive animations
- Marketing-style landing pages

Shrnk should feel understated and confident.

---

# Brand Personality

Shrnk is:

- Minimal
- Calm
- Fast
- Technical
- Reliable
- Clean

Shrnk is NOT:

- Playful
- Loud
- Flashy
- Experimental

---

# Logo

Always display the logo as:

shrnk

Rules:

- Lowercase only
- Text logo preferred
- Monochrome whenever possible
- Keep generous whitespace around the logo
- Never stretch or distort

# Typography

Preferred fonts:

- Geist
- Inter

Fallback:

- system-ui

Rules:

- Large bold headings
- Comfortable line height
- Medium weight buttons
- Minimal font size variations

Avoid decorative fonts.

---

# Layout

Desktop first.

Maximum content width:

1280px

Content should always breathe.

Large spacing is preferred over crowded layouts.

Use a consistent 8-point spacing system.

---

# Border Radius

Cards

16px

Buttons

12px

Inputs

12px

Badges

999px

---

# Shadows

Keep shadows minimal.

Prefer:

- borders
- subtle elevation

Avoid floating cards.

---

# Icons

Use only:

Lucide React

Keep icon size consistent:

18px–20px

Never mix icon libraries.

---

# Buttons

Primary

- Solid Indigo
- White text

Secondary

- Dark surface
- Thin border

Danger

- Red

Buttons should never use gradients.

---

# Cards

Cards should have:

- Dark background
- Thin border
- Soft radius
- No strong shadows

Every card should feel lightweight.

---

# Forms

Inputs should be:

- Large
- Comfortable
- High contrast

Focus state:

- Indigo border
- Soft ring

Never use bright blue browser defaults.

---

# Navigation

Navigation should remain simple.

Left

- Logo

Right

- Dashboard
- Shorten Link
- Profile
- Logout

Avoid deep navigation hierarchies.

---

# Landing Page

The landing page should be clean and product-focused.

Sections:

Hero

Features

How It Works

CTA

Footer

---

## Hero

Simple headline.

Example:

Shorten.
Manage.
Track.

Modern URL shortening for developers.

One supporting sentence.

Two buttons.

Primary CTA:

Get Started

Secondary CTA:

GitHub

No hero illustration.

No 3D graphics.

No AI artwork.

---

## Features

Three or four feature cards.

Examples:

Secure Authentication

Fast Redirect

Link Analytics

Simple Dashboard

Use icons only.

Avoid large illustrations.

---

## Dashboard

Dashboard is the heart of the application.

Priority order:

1. Quick Actions
2. Search
3. Link List
4. Analytics
5. Settings

The link table should always receive the most visual emphasis.

Statistics should remain compact.

Do not let metric cards dominate the page.

---

# Link Cards

Each link should display:

- Title
- Short URL
- Original URL
- Click Count
- Copy
- Analytics
- Edit
- Delete

Keep information compact.

Avoid oversized cards.

---

# Empty States

Empty pages should encourage action.

Example:

No links yet.

Create your first shortened URL.

Include one primary CTA.

---

# Loading States

Use skeleton loaders.

Avoid spinners whenever possible.

---

# Animations

Keep animations subtle.

Duration:

150–250ms

Use ease-out.

Never animate everything simultaneously.

No bounce effects.

---

# Responsive Design

Support:

Desktop

Tablet

Mobile

Navigation collapses on mobile.

Cards become stacked.

Tables become list layouts.

---

# Accessibility

Maintain sufficient color contrast.

All buttons require visible focus states.

Interactive elements must remain keyboard accessible.

---

# Overall Experience

A first-time visitor should think:

"This feels like a real developer SaaS."

The interface should look intentional, restrained, and production-ready.

When in doubt:

Choose simpler.
Choose cleaner.
Choose less.
